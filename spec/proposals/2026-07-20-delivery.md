# Change Proposal: portitor — release pipeline (binaries, signed, cross-arch)

## Context

portitor ships as source today: an operator building it runs `docker build` (which itself runs
`go build` inside the image) directly against a git checkout. There is no way to get a
`portitor` binary — for the container, or for running `add-role`/`validate-config`/`reconcile`
standalone on an operator's own machine — without cloning the repo and having a Go toolchain.
There is also no way to verify that a binary someone hands you actually corresponds to a given
commit: no version output, no checksum, no signature.

**Bootstrap note.** This proposal is deposited as a structural artifact — the spec record of the
change — not as a gate trigger. The normal spexmachina flow (`spex apply` → beads → an
orchestrated implementation pass through the portitor gate itself) is not usable for this change:
it depends on the very release tooling this proposal describes (the gate needs signed, versioned
releases of itself to be trustworthy infrastructure; that tooling doesn't exist yet). This slice
was implemented directly, by hand, once — with the spec structure preserved so it reads the same
as work that *did* go through the pipeline, and so the real pipeline (once portitor's own release
process is live) has a concrete precedent to work from rather than a green field.

## Proposed change

Add a GoReleaser-driven release pipeline, triggered by pushing a `v*` tag, that produces signed,
checksummed, cross-architecture binaries on GitHub Releases — and nothing else. No container
image is published (the Dockerfile remains a local build artifact); no package-manager formula;
no install script.

### 1. Cross-arch build (`.goreleaser.yaml`)

`linux`/`darwin` × `amd64`/`arm64` (four targets; no `windows` — portitor runs on the git server,
never on an agent's machine). `CGO_ENABLED=0`, `-trimpath`, and ldflags stamping `main.version`/
`main.commit`/`main.date` (new: `cmd/portitor/version.go`, plus a `version`/`--version`/`-v`
subcommand in `main.go`) from the tag/commit/build time, so a shipped binary's own `portitor
version` output is a direct, unfalsifiable link back to its provenance.

### 2. Signing + provenance

Every archive is minisign(Ed25519)-signed (`.goreleaser.yaml`'s `signs` block, key from the
`MINISIGN_SECRET_KEY` repository secret, written to a runner-local temp file for the job's
duration and removed after — never argv, never a logged command line) and covered by a SLSA build
provenance attestation (`actions/attest-build-provenance`). Both are independent of, not a
replacement for, sha256 checksums — three separate ways to trust a downloaded binary.

### 3. Machine-readable manifest

`scripts/generate-manifest.sh` turns GoReleaser's own `dist/artifacts.json` +
`dist/metadata.json` into a generic `manifest.json`
(`{schema_version, tool, version, git_sha, git_ref, built_at, status, artifacts:[…]}`), recomputing
each artifact's sha256/size from the archive bytes on disk rather than trusting an internal field
name. Published as a release asset so a downstream consumer (a Nix overlay, say) can pin a
per-target checksum programmatically. No bespoke delivery schema anywhere else — the executable
`.goreleaser.yaml` remains the one source of truth for how artifacts are actually built;
`manifest.json` is a derived, disposable projection of it.

### 4. CI tiering

Three trigger-scoped workflows: `ci.yml` (pull_request fast gate; push-to-main adds `-race`),
`fuzz.yml` (nightly, discovers every `func Fuzz*` in the tree at run time rather than a hardcoded
list — see `spec/reviews/2026-07-18.md` §5), `release.yml` (the `v*`-tag pipeline above).

## Non-goals

- No container image publication. The gate/egress containers are built locally by the operator
  from the repo `Dockerfile`; they are not release artifacts.
- No Homebrew/Scoop/AUR formula, no cross-repo checksum-dispatch notification (the beads_viewer
  and beads_rust templates this was adapted from both couple to infrastructure — a homebrew-tap
  repo, an ACFS installer-notification repo — that has no portitor equivalent). `manifest.json`
  gives a future consumer a stable integration point without portitor needing to know who they are
  in advance.
- No auto-migration or backward-compat shim for the release format itself; `schema_version` in
  `manifest.json` exists so a future incompatible change is a version bump a consumer can detect,
  not a silent shape change.

## Residuals (honest)

The pipeline was proven locally end-to-end (`goreleaser release --snapshot --clean`, a throwaway
minisign keypair, `minisign -Vm` / `sha256sum -c` / manifest cross-check against the real output —
see `spec/delivery/test_delivery.md`) but **never through actual GitHub Actions**: no tag has been
pushed, no workflow run has executed, `MINISIGN_SECRET_KEY` does not yet exist as a repository
secret. `RELEASE-SETUP.md` (repo root, not committed — scratch) is the checklist for the parts only
the repo owner can do: generate the real minisign keypair, set the secret, confirm Actions
permissions, and push a first `v0.1.0-rc.1` tag to observe a real run before `v0.1.0`.
