# Change Proposal: portitor — a git/landing gateway for agent pushes

## Context

Coding agents run in Docker containers and must be prevented from landing unwanted changes
(unsigned commits, pushes to a default branch, actions outside their role). A long design pass
established that **local, command-level controls cannot be a guarantee**: a command-string gate is
defeated by the unbounded ways a result can be produced — git plumbing (`commit-tree`,
`update-ref`), libgit2/pygit2, the GitHub REST API, and shell obfuscation (`eval`, base64, aliases).
`permissions.deny` globs and even a YubiKey touch policy can't enforce signing, because
`--no-gpg-sign` skips both the signature *and* the touch. Whatever the agent produces, it produces
*git objects and refs*; the only place those can be inspected reliably is the **receiving end**.

Three principles follow:

1. **Enforce at a chokepoint the agent can't cross, inspecting the *result*, not the command.** The
   chokepoint is a git server the agent must push through to make anything land.
2. **Identity is a credential, not a label.** "Which skill/role am I" is forgeable inside one
   session; a **signing key per role** is not. The gateway maps a commit's signer fingerprint to a
   role and enforces per-role rules.
3. **Capability removal beats interpretation.** The agent holds no credential to the real upstream;
   the gateway holds it. The agent's only path to landing is the gateway.

## Proposed change

Introduce **portitor** — a standalone, generic git-gateway (Go), deployed as its own container
between the agent and the real upstream (e.g. GitHub). The agent's `origin` is a bare repo on
portitor, reached over local SSH so `git-receive-pack` + `pre-receive` run under portitor's user
(the agent has no filesystem write to the bare repo). The agent has no upstream credential and
clones *from* portitor's mirror, so the only route to landing is through the gate.

### 1. `pre-receive` verification (this proposal's delivered slice)

On each push, git quarantines the incoming objects and runs `pre-receive` with the ref updates on
stdin (`<old> <new> <ref>`). portitor inspects the **objects/refs** and **rejects the whole push
atomically**, collecting *all* violations into one actionable `remote:` report so the agent can fix
everything in one pass. Initial checks, all on the result:

- **no push to / deletion of the default branch** (derived from the bare repo's HEAD, or configured).
- **every new commit signed by an allowed signer** — delegated to `git verify-commit` against an
  `allowed_signers` file portitor controls. Robust regardless of how the commit was created.

The crypto is git's; portitor orchestrates and decides. The gate is a pure function of
`(repo, ref-updates, config) → []violation`, so it is testable in isolation with ephemeral keys.

### 2. Identity → role (next)

portitor's `allowed_signers` maps each public key fingerprint to a **role** (implementer, fixer,
reviewer, you). Per-role rules become hard: e.g. a commit that closes a bead (a transition in
`.beads/issues.jsonl`) must be **reviewer- or owner-signed**. Each role runs in its own container
with exactly one forwarded no-touch key, so a container cannot sign as another role.

### 3. Forwarding (next)

On accept, `post-receive` forwards accepted **feature** branches to the real upstream using a
credential only portitor holds (the agent never sees it); the default branch is never forwarded
(PR/owner flow). GitHub-side actions (PR/comment/merge/close) are likewise expressed by the agent
as committed **intent artifacts** and executed by portitor with its own token — so the agent needs
no `gh` and no GitHub egress.

### 4. Content rules & ancestry (next)

`pre-receive` additionally enforces ancestry (a feature branch is based on current upstream main)
and pluggable **content rules** (e.g. valid bead-state transitions in `.beads/issues.jsonl`).
portitor stays generic — beads rules are config it loads, not built-in knowledge.

## Non-goals

- portitor does not orchestrate agents or workflows (that is the future **faber**, the evolution of
  `conductor`). portitor is purely the landing gate.
- portitor knows nothing of beads/spex; domain rules are external config.

## Residuals (honest)

The gate is unbypassable *for what lands* because it is the receiving server and the agent has no
other route. Local command-level layers (managed-settings denies, client git hooks) remain
honest-path feedback only. Exotic local mischief is contained by removing the agent's credentials
and network egress, not by inspecting commands.
