# Portitor — Combined Review & Remediation Plan (2026-07-18)

## 0. About this document

This is the consolidated remediation reference for portitor. It merges two review
passes:

- **Pass 1 (ledger)** — the four standard directions (input validation / robustness,
  failure-mode correctness, operator ergonomics, spec↔code fidelity).
- **Pass 2 (orthogonal)** — trust boundaries & untrusted-input handling, concurrency &
  races, invariants / property & fuzz testing, on-disk format evolution & migration.

Every finding below was verified against source (`file:line` anchors are exact against
`main` at the time of writing). Framing is neutral engineering throughout: correctness,
robustness, input validation, normalization, invariants, resource limits, cross-component
disagreement.

**Remediation scope = the union of both passes.** Several findings are *superseded* by the
redesigns in §1 (one coherent change closes a cluster of textual-scan defects); those are
labelled `superseded-by` rather than fixed line-by-line.

**How to read a finding:** `ID — title` · `file:line` · **Impact** · **Direction** · *Disposition*.

**Severity:** P1 = fix first · P2 = should fix · P3 = minor.
**Disposition:** `fix` (standalone change) · `superseded-by <group>` (folded into a redesign) ·
`spec-only` (code is correct, spec is stale) · `strong-point` (do not regress).

**Work order for the fix phase** (per the agreed chain: spec → review → fix, then
code → review → fix, then an independent conformance pass): the groups in §2 are ordered by
priority and by data-model dependency, not strictly by finding number. Group them by code
surface; do not re-touch the same function once per finding.

## 1. Design directions adopted

These were designed and agreed before this document. They define the target the fixes build
toward. Portitor's guiding principle is unchanged and reinforced: **portitor is a mechanistic
role×rule matcher; roles are opaque strings bound to signing-key fingerprints; domain knowledge
lives in config, not code** (mechanism-not-policy — the same split faber uses). The proposal's
three principles still hold: enforce at a chokepoint the agent can't cross, inspecting the
*result* not the command; identity is a credential (a signing key), not a label; capability
removal beats interpretation.

### 1.1 L1 — content-protection redesign (structural + semantic)

Replaces the single textual `RoleRule` (a regex over added diff lines) with two rule families.
This is the largest workstream and it supersedes the entire "textual scan" cluster.

**(a) Structural rules — native, portitor-computed.**
- Source: `git` name-status over the **full** diff with rename detection (`-M`/`-C`), *not* the
  pathspec-filtered diff (pathspec filtering breaks rename detection).
- Operations (file-level): `add`, `modify`, `delete`, `rename`.
- Rule = `path-glob × operation × role → allow/deny`.
- Global default is **allow** outside protected paths; a per-path `default` may override it;
  a role with a near-total restriction (e.g. a landing-only role that should author almost no
  commits) is expressed as an inverted rule. These role bindings are **config, not code**.
- Covers the ledger's "no path/pattern protection rules" gap (delete / rename / whole-file edit)
  directly, and is immune to the textual-scan evasions below because it reads name-status, not
  added text.

**(b) Semantic rules — parsing delegated to the consumer (`br`), transition computed generically.**
- The protected structured file (e.g. `.beads/issues.jsonl`) is parsed by **`br` itself**
  (`beads_rust`, the tool that owns the format), never by a portitor-internal parser. This
  eliminates the "textual check vs consumer parser" disagreement *by construction* — the parser
  that decides is the consumer.
- Hermetic invocation (verified empirically, br v0.2.16): portitor writes the git blob to a
  throwaway `.beads/issues.jsonl` and runs `br --no-db list --json`. No live DB, no config, no
  `init` needed; deterministic; **fail-closed on every hostile-input class tested** (invalid JSON,
  git conflict markers, control/NUL bytes, duplicate ids → a structured error envelope
  `.error.code`, non-zero exit, no partial output).
- Per commit: parse `parent:path` and `commit:path` via br → two record sets `{id → fields}`.
  Attribution is **per-commit** (each commit's signer must authorize the transitions that commit
  introduces).
- portitor then computes the transition **generically** over record deltas (all fields available;
  rules name the fields they care about). Two rule forms fall out of one primitive
  (`predicate(old, new) + role-effect + per-path default`):
  - *gate*: "transition T requires role ∈ S" (e.g. `status → closed` requires reviewer/merger).
  - *restrict*: "role R may only perform transitions ∈ S" (e.g. implementer only `open → in_progress`).
- **Build the general skeleton now, keep the vocabulary minimal.** General (expensive to retrofit,
  so build now): full-field record deltas, the `match/effect/default` rule model, a **versioned
  rule schema**, and a fixed precedence (per-path `default` + first-match / deny-wins). Minimal
  (cheap to extend additively later): the starter matcher set —
  `field {from/to/changed}`, `record {added/removed}`, `labels {contains/added}`, `role {in/not_in}`.
- **Fail-closed:** any br non-zero / error envelope, or any malformed protected `issues.jsonl`,
  rejects the whole push. Bonus invariant available for free: require the committed
  `issues.jsonl` to be br-valid.
- **Isolation of the br check:** subprocess-first — `exec.CommandContext` + timeout + memory
  rlimit + `io.LimitReader` on output + fixed argv; untrusted content reaches br only via a file
  / stdin, never a shell string. (A heavier sandbox is backlog — see §3.)

Pitfalls locked into the skeleton (cheap now, expensive later):
- Compare by **named fields**, not whole-record (br emits derived fields — `updated_at`,
  `compaction_level`, `dependency_count` — that would make every edit look "modified").
- The `restrict`-rule scope (only the named field vs the whole record) must be explicit in schema.
- Pin the br `--json` envelope via `br schema` + exit code, never human text.
- br's record schema and close semantics are a **br-version contract** (the toolset overlay pins
  v0.2.16); field/semantic drift is tracked via config + the format versioning of §1.3.

### 1.2 L2 — action mediation (`portitor pr`)

Portitor's action layer executes GitHub-side operations with its own credential after checking
authority (the agent holds no `gh` and no upstream token). Two parts.

**(a) Role×action policy → config.** Today `action.RoleCan` / `RoleActions` is a hardcoded
5-role allowlist (`internal/action/policy.go`). Move it to per-repo config with **default-deny**
(an action not listed for a role is refused — every action is privileged, unlike L1 content where
the global default is allow). The action *verbs* stay a closed mechanism set
(`comment|review|merge|close|fetch`); *who may* is config. Role-indifferent; closes the ledger's
"closed 5-role allowlist vs spec's free-form roles" mismatch.

**(b) Merge preconditions — re-derive from authoritative signals, do not trust the request**
(the discipline `MergeApproved` already uses for approval). All of it maps onto three `gh` fields
added to the existing `--json` query (`mergeStateStatus`, `mergeable`, `statusCheckRollup`,
`author`):
- `reviewDecision == APPROVED` (already enforced).
- `mergeStateStatus == CLEAN` (**mandatory**) — one field covers up-to-date-with-base (`BEHIND`),
  conflict-free (`DIRTY`), and blocking (`BLOCKED`). "Green on the merge *result*" ≈ up-to-date +
  required checks green (true "on the result" is a GitHub merge queue); a `BEHIND` branch is not
  mergeable even if its tip is green.
- required checks green via `statusCheckRollup` — **configurable** list (empty = advisory until CI
  exists on the repos).
- separation-of-duties — approver/merger ≠ PR commit author (mostly inherent since roles are
  distinct keys; the explicit check guards misconfiguration and an owner acting in multiple roles).
- **Enforcement is hybrid:** portitor re-derives for a clear policy verdict and an actionable error,
  but the final `gh pr merge` is the atomic gate (GitHub re-checks; a state change in the window
  fails the merge — TOCTOU is closed GitHub-side). Recommend the operator also enable GitHub
  branch protection (required checks + require-up-to-date) as defense-in-depth.

**(c) Audit.** A durable, server-side audit record of every L1 accept/reject and every L2 action
decision (who / what / repo / PR / verdict / reason). Closes the ledger's "no server-side audit
trail" item. In scope this pass.

Out of portitor's scope (a separate infra task): standing up real CI on the repos — L2 only gates
on its check-runs.

### 1.3 On-disk format versioning (Tier 1 + guards; migrations deferred)

- Add a **format-version stamp** to portitor's per-repo config (critical now that the L1 rule
  schema is a large new addition to that config) and to the hook shims.
- **Fail-closed direction is hard for the config:** a config whose version is higher than the
  running binary understands makes portitor **refuse to gate** — never gate with a partially
  understood ruleset. Silently dropping an unknown rule type is a weakened security control.
- The schema version is **independent of the application version** — bumped only when the on-disk
  format actually changes, so the guard trips only on a real format change.
- Hook shims carry a version marker and gain an idempotent `upgrade-repo` re-bake path (see
  Group F / FE-P7); the hook subcommand names become a frozen compatibility surface.
- Migrations / backward-compat readers are **deferred** (§3). Across a schema bump the operator
  re-bakes / re-provisions; there is no silent auto-migration.

### Group A — The gate verifies the real objects, deterministically (P1)

| ID | Title | Anchor | Disposition |
|----|-------|--------|-------------|
| PORT-1 | Ref-namespace unrestricted + gate git calls honor replace-objects | `internal/gate/gate.go:100-176`, `internal/git/git.go:19`, `gate.go:237` | fix |
| PORT-3 | Empty `AllowedSigners` falls back to ambient gitconfig (verdict env-dependent, can silently accept) | `internal/gate/gate.go:232-234` | fix |
| L-P2a | Ref/SHA fields from pre-receive stdin reach git without format validation | `cmd/portitor/main.go:552-567`, used `gate.go:210`, `forward.go:53` | fix |
| PORT-13 | `upstream_remote` / env reaches git argv un-`--`-guarded | `internal/gate/forward.go:53`, `main.go:432` | fix |
| L-P2b | No timeout/context on any subprocess (git/gh/ssh-keygen) | `internal/git/git.go:19`, `internal/action/action.go:23`, `addrole.go:216` | fix |
| PORT-12 | `newCommits` ignores `sc.Err()` → could accept unchecked commits | `internal/gate/gate.go:219-225` | fix |

**PORT-1** — `gate.Check` special-cases only the default branch and deletions; any other ref
namespace (`refs/replace/*`, `refs/notes/*`, `refs/tags/*`) is treated as a feature branch. A push
of `<G>:refs/replace/<sha(C)>` where `G` is already reachable is a create whose
`rev-list G --not --all` is empty → zero commits checked → accepted. Once that ref lands, every
gate/forward git subprocess (which runs plain `git`, no `--no-replace-objects` /
`core.useReplaceRefs=false`) reads the *substituted* object, so the `%G?` signature check, the
`git show <C>:path` content scan, and ancestry all verify `G` in place of `C`. **Direction:**
allowlist ref namespaces (only `refs/heads/<allowed>`; hard-refuse `refs/replace/*`, `refs/notes/*`),
run every gate/forward git call with `-c core.useReplaceRefs=false`, and forward only `refs/heads/*`.

**PORT-3** — `-c gpg.ssh.allowedSignersFile=` is passed only `if allowedSigners != ""`; with an
empty value git resolves the ambient `~/.gitconfig` / `/etc/gitconfig` / repo config
(`gh auth setup-git` writes one), so an unconfigured signer can verify as `G`. The doc claim
"empty ⇒ every commit untrusted" does not hold; the outcome can be *acceptance*. **Direction:**
pass `-c gpg.ssh.allowedSignersFile=<value>` unconditionally (empty pins fail-closed) or refuse to
run with empty `AllowedSigners`; consider `GIT_CONFIG_GLOBAL=/dev/null`-style isolation for the
verification call. (Distinct from the ledger's "silently-empty config" P2, which assumed
deterministic *rejection*.)

**L-P2a / PORT-13 / L-P2b / PORT-12** — validate SHAs (40-hex or zero) and `refs/`-prefix from
stdin; prefix rev args with `--`; validate the remote name shape (no leading `-`) and/or `--`-guard
it; add `CommandContext` + deadlines to git/gh/ssh-keygen (a hung subprocess blocks the push
indefinitely); check `sc.Err()` in `newCommits` (the loop the "every introduced commit is checked"
invariant rests on — its failure direction is *accepting* unchecked commits).

### Appendix findings — strong points (do NOT regress)

- Gate is fail-closed end to end: any internal error rejects the push; an unknown `%G?` status is a
  violation (`gate.go:141-147`); signature verdict parsing pads to three fields and treats any
  non-`G` as untrusted (`gate.go:238-242`).
- Config writes are atomic: temp-in-same-dir + fsync + rename + parent-dir fsync
  (`addrole.go:378-420`); the gate reads config as one atomically-renamed snapshot (`config.go:98`).
- SSH command routing is a strict allowlist; no `sh -c`; explicit argv everywhere
  (`main.go:348-372`, `git.go:19`). The L1 br check must preserve this discipline (untrusted content
  as data only).
- `parseUpdates` is fail-closed on scanner errors — oversized/garbage stdin rejects rather than
  truncates (`main.go:566`).
- The positional signer-line parse deliberately tolerates keytype-shaped principals and is
  regression-tested (`addrole.go:247-266`, `addrole_test.go:424`).
- `MergeApproved` is fail-closed for every non-`APPROVED` value including empty (`action.go:142`).
- Cross-file skew is fail-closed by construction: config is one snapshot, allowed_signers is re-read
  by git per commit, so mid-push add-role changes can only *add* violations, never a wrongful accept.

## 3. Deferred / backlog (retained, intentionally out of this pass)

- **L1 matcher vocabulary beyond the starter set:** regex on a field, numeric comparison
  (`priority`), cross-field conditions. Additive — a new match-type in the evaluator; existing
  configs keep working; no migration.
- **Sandbox isolation for the br check** (nsjail / bubblewrap / throwaway container) beyond the
  subprocess-first approach. Warranted only if a threat review of br-on-hostile-input requires it;
  the empirical probes (br fail-closes on malformed JSONL, runs no git, writes only under `.beads`)
  argue subprocess + timeout + rlimit + fail-closed is sufficient.
- **Whole-`.beads/` context** for semantic rules that need the dependency graph (today only the
  single `issues.jsonl` blob is fed to br).
- **Distinguishing richer "close" kinds** (e.g. `spex:obsolete` vs work-done) if the `labels`
  matcher proves insufficient. (`obsolete` vs `done` is a labels-layer convention over br, so the
  `labels` starter matcher covers the common case.)
- **`branch-delete` as a new L2 verb** (the release-bead cleanup flow wants it).
- **On-disk migrations / backward-compat readers** (Tier 2 of §1.3) — deferred until the upgrade
  model demands them.

**Pre-ship verification (belongs in the workstream's verify step, not the feature backlog):**
- br on very large input (memory / time) under rlimit + timeout.
- confirm `br --no-db` performs no network and no writes (run under no-network + read-only-fs).

## 5. Fuzz & property-test plan

No native fuzz targets or property tests exist today (tests are table/integration). Highest value:

1. `FuzzDiffAddsMatch` → after Group B, a `FuzzStructuralRules` / `FuzzTransitionEngine`: property
   that the structural verdict equals an independent name-status extractor's, and that the semantic
   verdict equals a direct comparison of br-parsed `parent`/`commit` record sets. Seeds: rename,
   delete, `-diff`/binary, NUL, conflict markers, duplicate ids, `++`-content.
2. **Matcher-vs-consumer agreement property** (regression for the confirmed L-CONF1 disagreement):
   drive a JSONL record serializer (random key order, whitespace, `\uXXXX` escaping) → rule fires
   ⇔ br parses the record as the protected transition. Should be green by construction after B.
3. `FuzzParseUpdates` — no panic; `err==nil` ⇒ round-trips; SHAs 40-hex-or-zero, refs `refs/`-prefixed.
4. `FuzzClassify` / `FuzzShellSplit` — no panic; kind "git" ⇒ exactly `[cmd, path]` with cmd in the
   closed table.
5. `FuzzConfigLoad` — no panic; `err==nil` ⇒ `Validate` doesn't panic; duplicate/unknown/case-collision
   keys rejected; version fail-closed.
6. `FuzzSignerLineKeyBlob` — keytype-shaped principals/comments never shift the extracted pair.
7. Determinism property: `gate.Check` under `GIT_CONFIG_GLOBAL=/dev/null` vs a poisoned global config
   → identical violations (pins PORT-3).
